==================================================
PHASE 1 — REAL TIME TRACKING
==================================================

Goal:
Turn the current time tracking from basic/partial into a real operational work timer.

Requirements:

1. Task work session model
Add/complete a dedicated work session / time log model with fields like:
- id
- taskId
- userId
- startedAt
- endedAt (nullable while active)
- durationMinutes
- status:
  - running
  - paused
  - completed
- pauseReason (nullable)
- createdAt
- updatedAt

2. Workflow behavior
- Technician can Start Work
- Technician can Pause Work
- Technician can Resume Work
- Technician can Stop / Complete work session when appropriate
- Only one active running session per task per user
- Prevent duplicate concurrent running sessions
- When paused/resumed, preserve previous duration correctly
- Total logged time must be aggregated correctly per task

3. UI
In Task Detail:
- show live timer while running
- show total logged time
- show session history
- show status badge:
  - Running
  - Paused
  - Completed
- add pause reason input when pausing
  examples:
  - waiting for parts
  - safety hold
  - access issue
  - inspection delay
  - other

4. Validation
- technician can only start/pause/resume their own assigned task unless operational roles are intentionally allowed
- cannot submit for QC while a running session still exists
- if task is approved/locked, time actions must be disabled

5. Dashboard metrics
Use actual logged data for:
- average completion time
- total logged hours
- active tasks in progress
- technician utilization (basic level)

==================================================
PHASE 2 — ATTACHMENTS (IMAGES / FILES)
==================================================

Goal:
Allow field users to attach evidence to maintenance tasks.

Requirements:

1. Attachment model
Add attachments table/model:
- id
- taskId
- uploadedByUserId
- fileName
- mimeType
- fileSize
- storageUrl
- attachmentType:
  - image
  - file
- createdAt

2. Upload support
Phase 1 scope:
- image upload required
- optional file upload support if easy
- support multiple attachments per task

3. Storage
Use a safe storage approach appropriate for the current environment.
If external object storage is not already configured:
- implement in a way that supports future migration to S3/R2
- do not hardcode an unsafe long-term hack
- keep storage abstraction clean

4. UI
In Task Detail:
- add "Attachments" section
- allow technician / engineer / supervisor to upload where permitted
- show image thumbnails
- click to preview full image
- allow deletion only by authorized roles or uploader if appropriate

5. Workflow usage
- technicians can attach before/after work evidence
- QC reviewer can inspect attachments during approval
- attachments must be visible in QC context

6. Validation
- limit size
- allow only safe mime types
- show clear upload errors
- do not silently fail

Track who did what and when across the task lifecycle.

Requirements:

1. Add audit log model
Fields:
- id
- taskId (nullable when needed)
- userId
- action
- entityType
- entityId
- detailsJson
- createdAt

2. Audit events to record
At minimum:
- task_created
- task_assigned
- task_started
- task_paused
- task_resumed
- task_submitted
- task_rejected
- task_revision_needed
- task_approved
- signature_saved
- attachment_uploaded
- attachment_deleted
- notification_created

3. Backend
Create a reusable helper:
- logAuditEvent(...)

Use it consistently across backend routes/services.

4. UI
In Task Detail:
- add "Audit Timeline" or "Activity Log" section
- show:
  - action label
  - actor name
  - timestamp
  - optional details summary

5. Readability
Make audit entries human-readable, not raw JSON dumps
