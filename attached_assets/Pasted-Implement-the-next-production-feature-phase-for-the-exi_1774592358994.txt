========================================
PHASE A — ELECTRONIC SIGNATURE
========================================

Goal:
Allow technicians and supervisors to sign tasks electronically in a structured, auditable way.

Requirements:

1. Add signature support to task execution and QC approval flow
- Technician can add signature before final Submit
- Supervisor/Engineer can add signature when approving QC
- Store:
  - signer name
  - signer role
  - taskId
  - signedAt
  - signature image data or stored file path
  - signature type:
    - technician_completion
    - supervisor_qc_approval

2. UI
- Add a signature pad using a canvas-based component
- Buttons:
  - Clear
  - Save Signature
- Show saved signature preview
- Show signature timestamp and signer identity
- In Task Detail:
  - technician signature section
  - supervisor approval signature section
- Approved tasks should display signatures in read-only mode

3. Validation rules
- Technician signature required before Submit
- Supervisor signature required before final Approve
- Reject does not require signature
- Prevent duplicate conflicting signatures of the same type unless explicitly replaced

4. Backend
- Add signatures table/model
- Add endpoints:
  - save task signature
  - get task signatures
  - replace signature (optional, controlled)
- Ensure auth user identity is used, not client-supplied signer identity
- Include signatures in task detail response

5. Audit
- Log signature creation / replacement in audit logs if audit log exists
- If no audit log table exists yet, create a minimal audit log implementation for signature actions only

========================================
PHASE B — NOTIFICATIONS
========================================

Goal:
Add in-app notifications tied to the real workflow.

Requirements:

1. Notification types
- task_assigned
- task_submitted
- task_rejected
- task_revision_needed
- task_approved
- task_overdue

2. Trigger rules
- When a task is assigned:
  -> notify assigned technician
- When technician submits task:
  -> notify relevant supervisor/engineer users
- When QC rejects:
  -> notify assigned technician
- When QC approves:
  -> notify assigned technician and creator if different
- When task becomes overdue:
  -> notify assigned technician + supervisory users

3. Delivery scope
Phase 1:
- In-app notifications only
- Notification bell/icon in the layout header/sidebar
- Unread count badge
- Notification list page or slideout panel
- Mark as read
- Mark all as read

4. Notification data model
Store:
- id
- userId
- taskId (nullable if needed)
- type
- title
- message
- isRead
- createdAt

5. UI requirements
- Clean Siemens-style light theme
- unread badge visible
- grouped by Today / Earlier if easy
- clicking a notification navigates to related task detail when applicable

6. Overdue behavior
- Use existing computed overdue logic
- Add a safe mechanism so overdue notifications do not spam repeatedly
- Only notify once per task overdue event unless status changes and becomes overdue again

========================================
PHASE C — ROLE PERMISSIONS
========================================

Goal:
Enforce real operational permissions based on user role.

Roles:
- site_manager
- engineer
- supervisor
- technician

Requirements:

1. Permission rules
Technician:
- view only assigned tasks (or tasks relevant to them)
- start / pause / resume work
- add notes
- add execution checklist data
- add technician signature
- submit task
- cannot approve/reject QC
- cannot create tasks unless explicitly allowed later

Supervisor:
- view all operational tasks
- review submitted tasks
- approve / reject QC
- sign QC approval
- view notifications
- may create tasks

Engineer:
- create tasks
- assign technicians
- review tasks
- approve / reject QC
- sign QC approval
- view dashboards/reports
- view notifications

Site Manager:
- broad visibility across all tasks
- create tasks
- assign tasks
- review and approve
- full dashboard/report visibility
- user/operations oversight

2. Backend authorization
- Do not rely on frontend checks only
- Add server-side authorization middleware/helpers
- Protect endpoints by role
Examples:
- create task -> engineer / supervisor / site_manager
- QC approve/reject -> engineer / supervisor / site_manager
- start/pause/resume/submit -> technician on own assigned task, or authorized operational roles if needed
- notifications -> only for authenticated user’s own notifications unless manager/admin access is intentionally added

3. Frontend permission gating
- Hide or disable actions user cannot perform
- Examples:
  - technician should not see approve/reject buttons
  - technician should not see create-task action if not allowed
  - supervisor/engineer should see QC controls
- AppLayout can show user role clearly

4. Data seeding
Add realistic demo users for:
- 1 site manager
- 1 engineer
- 1 supervisor
- 2 technicians
